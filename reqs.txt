---

### 1. Overview  
This document describes the complete tenant (doctor/clinic owner) sign‚Äëup, onboarding, and initial setup flow. It covers account creation, email verification, subscription selection, organization structure setup, staff user creation, and subsequent role management via RBAC. No implementation code is provided; the steps are purely a requirements specification for the development team.

### 2. Data Models Involved  

| Model | Purpose |
|-------|---------|
| `profiles` | Stores tenant account metadata (owner profile). One record per tenant. |
| `subscription_plans` | Available plans configured by the app owner. |
| `subscription_features` | Features linked to each plan. |
| `tenant_subscriptions` | Active subscription for a tenant (plan, start/end dates). |
| `branches` | Physical branches belonging to a tenant. |
| `clinics` | Clinics operating under a branch. |
| `roles` / `permissions` (RBAC module) | Roles and permissions that a tenant can assign to their users. |

All steps assume an authenticated session after sign‚Äëup / email verification.

### 3. Step‚Äëby‚ÄëStep Flow  

#### A. Sign‚ÄëUp & Immediate Profile Creation  

1. **User initiates sign‚Äëup** with email and password using the authentication system (Supabase Auth).  
2. **Immediately after successful auth sign‚Äëup**, a background operation must **create a record in the `profiles` table** for this tenant. The record contains at least the user‚Äôs unique ID and marks the account as incomplete/onboarding required.  
   - If profile creation fails, the sign‚Äëup should be rolled back or the user notified to retry.  
3. **Fix the email redirect URL**: The confirmation email‚Äôs `emailRedirectTo` parameter must point to the **onboarding screen**, not `/auth/callback`.  
   - Example target: `https://yourapp.com/onboarding` (or a deep link).  
   - This change is made in the authentication configuration or in the API call that sends the confirmation email.

#### B. Email Verification & Onboarding Entry  

4. **User clicks the confirmation link** in the email and is redirected to the onboarding screen.  
5. **Onboarding screen guard**:  
   - Verify that a `profiles` record exists for the currently authenticated user.  
   - If no profile exists, display an error and do not allow progress (the sign‚Äëup profile creation step failed or was bypassed).  
6. If the profile exists, **proceed to subscription selection**.

#### C. Subscription Plan Selection  

7. **Fetch available subscription plans** from the `subscription_plans` table (as configured by the app owner).  
8. For each plan, display its associated **features** from `subscription_features`.  
9. **User selects a plan** and confirms.  
10. **Create a row in the `tenant_subscriptions` model**:  
    - `start_date` = current date/time.  
    - `end_date` = `start_date` + plan duration **in months** (e.g., 1 month, 12 months). Month‚Äëaddition logic must handle end‚Äëof‚Äëmonth gracefully.  
    - Store the selected plan reference and tenant ID.  
11. Update the tenant‚Äôs profile record to reflect an active subscription (e.g., status flag).

#### D. Branch & Clinic Setup  

12. **After subscription**, guide the tenant to set up their organization structure.  
13. **Add branches**:  
    - The tenant can create one or more records in the `branches` table.  
    - Each branch belongs to the current tenant (`tenant_id`).  
    - Basic branch details (name, address, etc.) are captured.  
14. **Add clinics**:  
    - For each branch, the tenant can create records in the `clinics` table.  
    - Each clinic is linked to a branch and implicitly to the tenant.  
15. This step may be optional; a ‚Äúskip for now‚Äù button should be available, with the ability to return later.

#### E. Adding Staff Users  

16. **Tenant can add users** (e.g., other doctors, assistants) to their tenant:  
    - Provide an interface to enter the new user‚Äôs email (and optionally a name).  
17. **Create the user account** using the **Supabase Auth Admin API** (server‚Äëside).  
    - Generate a temporary password or send an invite email automatically.  
    - **Assign the default role** `user` in the authentication system (if role metadata is supported).  
18. **Create a corresponding profile** in the `profiles` table for this new user:  
    - `is_owner` = **false**.  
    - `tenant_id` = the ID of the currently logged‚Äëin tenant (the owner).  
    - The user‚Äôs auth ID must be linked to this profile.  
19. The new user will then complete account setup (e.g., set password) via the standard email invitation flow.

#### F. RBAC ‚Äì Role & Permission Management  

20. **After onboarding**, the tenant (owner/doctor) can access an RBAC module to manage their staff:  
    - View all users belonging to their tenant.  
    - Change a user‚Äôs **role** (e.g., from `user` to `manager`, `doctor`, `receptionist`).  
    - Assign or revoke specific **permissions** (as defined in the RBAC system).  
21. These changes are scoped **only to users within the tenant**; the tenant cannot modify other tenants‚Äô users.  
22. The RBAC system must ensure that role updates are reflected in both the auth system and the application‚Äôs authorization checks.

### 4. Edge Cases & Error Handling  

- **Profile creation fails during sign‚Äëup**: Transactional handling ‚Äì if profile insert fails, delete the auth user and show a descriptive error.  
- **Onboarding accessed without profile**: Redirect to an error page or prompt re‚Äësign‚Äëup.  
- **Subscription creation failure**: Roll back any profile updates; allow retry.  
- **Email redirect link broken**: Ensure the `emailRedirectTo` is correctly set and the onboarding route handles the authenticated state properly.  
- **Date calculation for month‚Äëbased subscriptions**: Use a library/function that adds months accurately, accounting for months with fewer days (e.g., Jan 31 + 1 month ‚Üí Feb 28).  
- **Staff user creation failure**: If Auth Admin API fails, do not create the profile; show a clear error and allow retry.  
- **RBAC assignment**: Prevent the owner from removing their own admin privileges unintentionally.
